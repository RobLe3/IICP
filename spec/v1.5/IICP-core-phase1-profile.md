**Version**: 1.0.0  
**Date**: 2026-05-14  
**Status**: accepted  
**Authority**: Protocol Steward  
**Relates to**: SPEC_ANALYSIS.md §1 and §7, GitHub issue #13

## Purpose

This document defines the minimal IICP implementation required for iicp.network Phase 1 (Proof of Concept). It is a subset of IICP v1.4.2 and is the authoritative contract for Design Gates on issues #1 (directory), #2 (adapter), and #3 (proxy).

Any implementation that satisfies all MUST requirements in this profile is **IICP-Core Phase 1 compliant**.

---

## 1. Message Subset

Phase 1 implements 7 of 14 IICP message types. The remaining 7 are explicitly deferred.

### 1.1 In-scope messages (Phase 1)

| IICP Message | Phase 1 Form | Notes |
|-------------|-------------|-------|
| INIT | `POST /v1/register` | Node → Directory registration |
| ACK | Registration response `{node_token}` | Directory → Node |
| CALL | `POST /v1/task` | Proxy → Adapter task submission |
| RESPONSE | Task response `{task_id, status, result, metrics}` | Adapter → Proxy |
| PING | `POST /v1/heartbeat` | Node → Directory liveness signal |
| PONG | Heartbeat response `{ok: true}` | Directory → Node |
| CLOSE | HTTP connection close | Implicit — no explicit CLOSE message in Phase 1 |

### 1.2 Deferred messages

| IICP Message | Deferred to | Reason |
|-------------|------------|--------|
| DISCOVER | Phase 2 | Phase 1 uses directory REST; P2P DISCOVER is mesh-only |
| SUB_PROTOCOL | Phase 2 | MCP binding requires formal binding spec (#15) |
| FEEDBACK | Phase 2 | Reputation system not needed for PoC |
| CONTROL | Phase 3 | Elevated privileges — complex trust model required |
| ADVERTISE | Phase 2 | QuDAG-specific mesh advertisement |
| OBSERVE | Phase 2 | QuDAG-specific |
| TELEMETRY | Phase 2 | QuDAG-specific |

## 2. Field Subset — Registration (INIT → POST /v1/register)

### 2.1 Required fields (MUST)

| Field | Type | Constraint |
|-------|------|-----------|
| `endpoint` | string (URL) | HTTPS only; validated by liveness check before token issued |
| `region` | string | IANA-style tag (e.g., `us-west`, `eu-central`); max 64 chars |
| `capabilities` | array | At least one capability; each MUST include `intent` |
| `capabilities[].intent` | string (URN) | MUST match `urn:iicp:intent:<domain>:<action>:v<N>` format |
| `capabilities[].models` | array of strings | At least one model name |
| `capabilities[].max_tokens` | integer | > 0 |
| `limits.max_concurrent` | integer | 1–256 |
| `limits.tokens_per_min` | integer | > 0 |

### 2.2 Optional fields (SHOULD in Phase 1)

| Field | Type | Notes |
|-------|------|-------|
| `node_id` | string (UUID) | Client MAY provide; directory generates if absent |
| `public_key` | string (base64) | Ignored in Phase 1; reserved for Phase 2 signing |
| `availability` | array | Time windows; used in scoring (ADR-008) |
| `availability[].start` | string (HH:MM) | UTC |
| `availability[].end` | string (HH:MM) | UTC |
| `availability[].share` | float (0-1) | Fraction of capacity during window |

### 2.3 Deferred fields (NOT in Phase 1)

| Field | Deferred to | Reason |
|-------|------------|--------|
| `auth_did` | Phase 3 | DID identity not yet required |
| `sub_protocol` / `sub_protocol_version` | Phase 2 | MCP binding (#15) |
| `adaptive_lock` | Phase 3 | Resource locking |
| `X-IICP-Transport-Hint: qudag` | Phase 2 | Only HTTPS in Phase 1 |
| `X-IICP-Auth-Method: did` | Phase 3 | DID |
| `telemetry_interval_ms` | Phase 2 | QuDAG-specific |

### 2.4 Registration response (ACK)

```json
{
  "node_id": "uuid",
  "node_token": "opaque-bearer-token",
  "expires_at": null,
  "directory": "https://iicp.network"
}
```

`node_token` is a 32-byte random value, returned once in plaintext, stored hashed (bcrypt) by directory. The client MUST store this securely.

## 3. Field Subset — Task (CALL → POST /v1/task)

### 3.1 Required fields (MUST)

| Field | Type | Constraint |
|-------|------|-----------|
| `task_id` | string (UUID v4) | Generated by proxy; checked for uniqueness at adapter |
| `intent` | string (URN) | MUST match registered capability intent |
| `payload` | object | Intent-specific; passed through to backend |
| `constraints.timeout_ms` | integer | > 0; adapter MUST abort if exceeded |
| `auth.node_token` | string | Bearer token; validated by adapter on every request |

### 3.2 Optional fields (SHOULD in Phase 1)

| Field | Type | Notes |
|-------|------|-------|
| `constraints.qos` | string | `interactive` \| `batch` \| `best-effort` |
| `trace.trace_id` | string (16-byte hex) | Propagated for observability |
| `trace.origin_node` | string (UUID) | Identifies the proxy/client |

### 3.3 Deferred fields (NOT in Phase 1)

| Field | Deferred to | Reason |
|-------|------------|--------|
| `mem_handle` / `mem_lock` | Phase 3 | Resource locking |
| `chunk_seq` | Phase 3 | Chunked streaming |
| `sig_agent` | Phase 2 | Signed requests (HMAC) |
| `X-IICP-Lock` | Phase 2/3 | Resource locking |
| `X-IICP-Trace-Hash` | Phase 2 | Cryptographic trace integrity |
| `peer_id` | Phase 2 | Direct peer routing (not directory-mediated) |
| `billing.*` | Phase 3 | Credits system |

### 3.4 Task response (RESPONSE)

```json
{
  "task_id": "uuid",
  "status": "success | error | timeout",
  "result": { },
  "metrics": {
    "latency_ms": 320,
    "tokens_used": 120
  },
  "error": null
}
```

On error: `status = "error"`, `result = null`, `error = {"code": "...", "message": "..."}`. MUST NOT include stack traces or internal paths.

## 4. Field Subset — Heartbeat (PING → POST /v1/heartbeat)

### 4.1 Required fields (MUST)

| Field | Type | Notes |
|-------|------|-------|
| `node_id` | string (UUID) | Must match a registered node |
| (Authorization header) | Bearer node_token | Validated by directory |

### 4.2 Optional fields (SHOULD)

| Field | Type | Notes |
|-------|------|-------|
| `load` | float (0-1) | `active_jobs / max_concurrent` |
| `active_jobs` | integer | Current concurrent tasks |
| `available` | boolean | Whether node is accepting tasks |

### 4.3 Heartbeat response (PONG)

```json
{ "ok": true, "next_heartbeat_ms": 30000 }
```

## 5. Discovery (Phase 1 form — directory REST)

Discovery in Phase 1 is a directory REST query, not the IICP DISCOVER message (which is peer-to-peer, Phase 2).

### Query

```
GET /v1/discover?intent=urn:iicp:intent:llm:chat:v1&qos=interactive&region=eu-central&limit=10
```

| Parameter | Required | Notes |
|-----------|---------|-------|
| `intent` | MUST | Full URN |
| `qos` | SHOULD | `interactive` \| `batch` \| `best-effort` |
| `region` | MAY | Filter preference, not exclusion |
| `limit` | MAY | Default 10, max 50 |

### Response

```json
{
  "nodes": [
    {
      "node_id": "uuid",
      "endpoint": "https://node1.example.com",
      "score": 0.91,
      "latency_estimate_ms": 120,
      "available": true,
      "region": "eu-central"
    }
  ],
  "count": 1,
  "query_ms": 12
}
```

Only nodes with `available = true` AND `last_seen` within 90 seconds are returned. Score computed per ADR-008. Score < 0.1 → excluded.

## 6. Transport and Encoding

| Aspect | Phase 1 | Deferred |
|--------|---------|---------|
| Encoding | JSON | CBOR (Phase 3) |
| Transport | HTTPS/1.1 or HTTPS/2 | QUIC (Phase 3), QuDAG (Phase 2) |
| TLS version | TLS 1.3 MUST | — |
| Auth | Bearer node_token | JWT (Phase 2), DID (Phase 3), PQ (Phase 4) |
| Framing | REST over HTTP | Binary IICP framing (Phase 3) |

## 7. Liveness and Expiry

| Rule | Value | Authority |
|------|-------|----------|
| Heartbeat interval | 30 seconds | Adapter sends |
| Node expiry threshold | 90 seconds since last heartbeat | Directory enforces (ADR-008) |
| Expiry check frequency | Every 30 seconds (cron) | Directory scheduler |
| Node re-activation | On next valid heartbeat | Automatic |

---

## 8. Error Codes (Phase 1)

| Code | HTTP | Meaning |
|------|------|---------|
| `unauthorized` | 401 | Missing or invalid node_token |
| `forbidden` | 403 | Token valid but insufficient rights |
| `not_found` | 404 | node_id not in registry |
| `conflict` | 409 | Duplicate task_id (idempotency — ADR-010) |
| `rate_limited` | 429 | Registration rate limit exceeded |
| `capacity_exceeded` | 429 | max_concurrent reached at adapter |
| `task_timeout` | 504 | Task exceeded constraints.timeout_ms |
| `backend_error` | 502 | Inference backend failed (details not exposed) |
| `validation_error` | 422 | Input schema validation failed |

---

## 9. Security Requirements (Phase 1 minimum)

Per SECURITY.md and ADR-006:

- TLS 1.3 MUST be enforced on all endpoints
- `POST /v1/register` rate limit: 10/min per IP (MUST)
- node_token: 32+ bytes cryptographically random, stored hashed (bcrypt), returned once (MUST)
- Adapter MUST validate node_token on every `POST /v1/task`
- Error responses MUST NOT include stack traces, file paths, or DB structure
- Task payloads MUST NOT appear in logs (task_id only)
